  28	- At least 3 of 5 versions configure cleanly enough to pass a structural build check

  30	### Context

  32	1. Nested //? directives inside /* */ blocks confusing the preprocessor
  33	2. Interaction between Stonecutter's source distribution and Gradle's multi-project setup
  34	3. Version-specific incompatibility (try upgrading to Stonecutter 0.7+ if available)
  35	
  36	**Evidence:**
  37	```
  38	java.lang.Exception: Failed to switch from 1.21.4 to 1.21.1 (and similar for 1.20.1).
  39	Direct :sub:build commands succeed structurally but show NO-SOURCE on every Kotlin task
  40	```
  41	
  42	### Suggested approach
  43	1. Audit //? directive structure in AttributeCompat.kt and DndWeaponItem.kt for nesting depth
  44	2. If nesting is the issue, flatten //? blocks or use separate files per version branch
  45	3. Test Stonecutter 0.7-SNAPSHOT (if available) to verify upward compatibility
  46	4. Fallback: implement git-branches-per-version strategy (design doc Section 7)

 115	## Task 4: Resolve FLK version mismatch for MC 1.20.1
 116	
 117	**Files:** versions/1.20.1/gradle.properties
 118	
 119	**From bug:** wave-11-agent-1-bug-5 (P3, incorrect_implementation)
 120	
 121	### Acceptance criteria
 122	- versions/1.20.1/gradle.properties specifies a valid FLK version (1.10.20+kotlin.1.9.24)
 123	- :1.20.1:build completes (may show runtime warnings about Kotlin stdlib version mismatch)
 124	- Validate at runtime: no ClassCastException or NoSuchMethodError from Kotlin stdlib conflicts
 125	
 126	### Context
 127	The plan-specified FLK version 1.10.20+kotlin.2.0.21 does not exist. FLK 1.10.x line supports only Kotlin 1.9.x. Using FLK 1.10.20+kotlin.1.9.24 (the latest 1.10.x) but introducing a potential stdlib conflict:
 128	- Kotlin compiler bundled with our Kotlin plugin = 2.3.21
 129	- FLK 1.10.20+kotlin.1.9.24 embeds Kotlin 1.9.24 stdlib
 130	
 131	**Evidence:**
 132	```
 133	Maven 404. The 1.10.x FLK line ships only with Kotlin 1.9.x bundled stdlib
 134	```
 135	
 136	### Suggested approach
 137	1. Set versions/1.20.1/gradle.properties: flk_version=1.10.20+kotlin.1.9.24
 138	2. Build :1.20.1:build and monitor for Kotlin stdlib conflicts
 139	3. If runtime conflicts occur, attempt FLK 1.13.x+ (requires downgrading Kotlin compiler to 1.9.24 for 1.20.1 only via Stonecutter)
 140	4. Document known limitation: 1.20.1 uses legacy Kotlin 1.9; verify backward-compat testing on 1.20.1 recipes/registries

 144	## Task 5: Validate and fix longsword recipe JSON schema
 145	
 146	**Files:** src/main/resources/data/dndweapons/recipe/longsword.json (and per-version overrides)
 147	
 148	**From bug:** wave-7-agent-1-bug-5 (P2, incorrect_implementation)
 149	
 150	### Acceptance criteria
 151	- longsword.json loads without "Map entry I" errors on all applicable MC versions
 152	- Recipe returns a dndweapons:longsword item with count=1
 153	- Datapack reload log shows no recipe parse errors
 154	
 155	### Context
 156	The shared recipe uses Fabric's `tag:c:ingots/iron` ingredient syntax, which requires a `fabric:type` discriminator in MC 1.21.4+ but not in 1.20.1.
 157	
 158	**Evidence:**
 159	```
 160	Server log: 'Map entry I : Input does not contain a key [fabric:type]: MapLike[{tag:c:ingots/iron}]'
 161	```
 162	
 163	### Suggested approach
 164	1. Test both schemas:
 165	   - **Schema A (1.20.1-compatible):** Use raw `minecraft:ingredient` with nested tag reference
 166	   - **Schema B (1.21+):** Add `fabric:type: "tag"` discriminator
 167	2. Create version-specific recipe overrides:
 168	   - versions/1.20.1/src/main/resources/data/dndweapons/recipes/longsword.json (Schema A)
 169	   - src/main/resources/data/dndweapons/recipes/longsword.json (Schema B for 1.21+)
 170	3. Run datapack reload on 1.20.1 and 1.21.4 instances to validate
 171	4. Document accepted ingredient format per MC version in code comments
