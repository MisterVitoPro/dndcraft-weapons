### [P0-001] Seven ranged DnD weapons (blowgun, sling, hand_crossbow, heavy_crossbow, longbow, musket, pistol) are never registered as items
**Confidence:** confirmed | **Corroborated by:** 2 agents (qa-data-flow, qa-architecture)
**Location:** src/main/kotlin/com/dndweapons/catalog/Weapons.kt:441 (filter) + src/main/kotlin/com/dndweapons/registry/WeaponRegistrarImpl.kt:35-45 (registerAll)
**Description:** `Weapons.ALL_TIERED` filters by `it.ranged in setOf(RangeKind.NONE, RangeKind.THROWN)`, excluding 7 non-vanilla-mapped ranged weapons (BLOWGUN, SLING, HAND_CROSSBOW, HEAVY_CROSSBOW, LONGBOW, MUSKET, PISTOL — all have ranged = BOW/CROSSBOW/FIREARM/SLING/BLOWGUN). `WeaponRegistrarImpl.registerAll(entries)` only iterates `entries` (which is ALL_TIERED) for tiered specs, plus a manual loop over `Weapons.ALL` for *vanilla-mapped* specs. There is NO loop for non-vanilla-mapped non-melee/thrown ranged weapons. Consequence: `BuiltInRegistries.ITEM` has no entry for `dndweapons:blowgun`, etc. `AcquisitionCatalog.validate()` logs errors at startup; loot pools and villager trades silently produce no offers (FLOW-002, FLOW-003); the model/recipe/texture/lang assets for these 7 weapons are dead. Master spec §4 lists all 6 martial ranged + Dart + Sling as `registered` (Dart's THROWN tier means it IS registered correctly).
**Evidence:**
```kotlin
// Weapons.kt:440-442
val ALL_TIERED: List<Pair<WeaponSpec, Tier>> = ALL
    .filter { it.vanillaRoleTag == null && it.ranged in setOf(RangeKind.NONE, RangeKind.THROWN) }
    .flatMap { base -> Tier.values().map { tier -> base.atTier(tier) to tier } }

// WeaponRegistrarImpl.kt:35-45
override fun registerAll(entries: List<Pair<WeaponSpec, Tier>>) {
    for (spec in Weapons.ALL) {
        if (spec.isVanillaMapped) register(spec, Tier.IRON)   // only vanilla-mapped
    }
    for ((spec, tier) in entries) register(spec, tier)        // only ALL_TIERED (no ranged non-vanilla-mapped)
}
```
**Suggested fix:** Add a third loop in `WeaponRegistrarImpl.registerAll`:
```kotlin
for (spec in Weapons.ALL) {
    if (!spec.isVanillaMapped && spec.ranged !in setOf(RangeKind.NONE, RangeKind.THROWN)) {
        register(spec, Tier.IRON)   // ranged weapons get IRON tier only (no smithing ladder per Phase 4 §1)
    }
}
```
**Related files:** acquisition/AcquisitionCatalog.kt (references these IDs), acquisition/WeaponLookup.kt (returns null), trade/WeaponTradeRegistrar.kt:131 (buildOffer returns null), codegen/Phase5TradeCodegen.kt:46 (emits trade JSONs for unregistered items)

### [P0-003] Shared smithing-transform recipes use MC 1.21.5+ string shorthand format; 1.21.4 has NO overlay so may reject them
**Confidence:** likely | **Corroborated by:** 1 agent (qa-correctness)
**Location:** src/main/resources/data/dndweapons/recipe/longsword_diamond.json (and all 53 other smithing-transform recipes)

```json
{
  "type": "minecraft:smithing_transform",

  "type": "minecraft:smithing_transform",

  "addition": { "tag": "c:gems/diamond" },
  "result": { "id": "dndweapons:longsword_diamond" }
}
```
**Suggested fix:** Convert the shared smithing-transform recipes to object form (universally accepted across 1.21.1+). Drop the 1.21.1 overlay smithing recipes once the shared format matches. ### [P1-005] PlayerAttackMixin uses @ModifyArg + ThreadLocal sweep-guard; Phase 3 spec sketched @ModifyVariable — undocumented design deviation
**Confidence:** confirmed | **Corroborated by:** 2 agents (qa-correctness, qa-architecture)
**Location:** src/main/kotlin/com/dndweapons/mixin/PlayerAttackMixin.kt:58-119
**Description:** Phase 3 spec §3 sketches `@ModifyVariable` against the local damage `float f` in `Player.attack()`. The implementation uses three mixin methods: `@Inject(HEAD)` to capture the primary target into a `ThreadLocal`, `@Inject(RETURN)` to clear it, and `@ModifyArg(index=1)` on the `hurt`/`hurtOrSimulate` invocation that consumes the ThreadLocal flag to limit modifier application to the first hit (not sweep targets). The implementation is functionally correct and likely MORE robust than the spec sketch (immune to LVT shifts across versions), but: (a) the architectural decision to use a sweep guard is undocumented, (b) the spec's "verify mixin injects successfully on every version" acceptance criterion now needs to cover three injection sites instead of one, and (c) the ThreadLocal field on each Player instance has no exception-path cleanup — if `Player.attack` throws between HEAD inject and RETURN, the ThreadLocal retains the stale Entity reference until the thread next executes Player.attack.
**Evidence:** PlayerAttackMixin.kt lines 56-119 show three mixin methods + ThreadLocal field. Phase 3 spec §3 shows single `@ModifyVariable`.
**Suggested fix:** Document the design deviation in Phase 3 spec or an inline architectural note explaining: (1) why @ModifyArg was chosen over @ModifyVariable, (2) what the sweep-entity problem is and why a guard is needed, (3) why the spec sketch is insufficient. Change `@Inject(at = RETURN)` to `@Inject(at = TAIL)` to fire on both normal and exception exits, eliminating the ThreadLocal leak window.
**Related files:** docs/superpowers/specs/2026-05-17-dnd-weapons-phase-3-design.md §3

### [P1-007] fabric.mod.json's `java`, `minecraft`, `fabricloader`, and `fabric-language-kotlin` constraints are too loose for the per-version build matrix
**Confidence:** likely | **Corroborated by:** 1 agent (qa-config-env)
**Location:** src/main/resources/fabric.mod.json:25-31
**Description:** The shared `fabric.mod.json` hardcodes `"java": ">=17"`, `"minecraft": ">=1.20.1"`, `"fabricloader": ">=0.16.0"`, `"fabric-language-kotlin": ">=1.10.0"`. ```
**Suggested fix:** Add `mc_version_range`, `flk_min_version`, `loader_min_version` to each per-version gradle.properties. Token-expand them into fabric.mod.json via `processResources`'s `expand()` block. Each jar then declares accurate per-version dependency constraints.
