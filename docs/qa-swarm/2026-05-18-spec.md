## P0 — Critical (must fix before any further phase)

### P0-001 Register the 7 missing ranged DnD weapons

**Root cause:** `Weapons.ALL_TIERED` filters out non-NONE/non-THROWN ranged weapons. `WeaponRegistrarImpl.registerAll` does not loop them.

**File:** `src/main/kotlin/com/dndweapons/registry/WeaponRegistrarImpl.kt`

**Change:** After the existing vanilla-mapped loop, add a non-tiered iron-only loop for ranged non-vanilla-mapped weapons.

```kotlin
override fun registerAll(entries: List<Pair<WeaponSpec, Tier>>) {
    // 1. Vanilla-mapped specs (Shortsword, Shortbow, Light Crossbow, Trident).
    for (spec in Weapons.ALL) {
        if (spec.isVanillaMapped) register(spec, Tier.IRON)
    }
    // 2. NEW: Ranged non-vanilla-mapped specs (bow/crossbow/firearm/sling/blowgun).
    //    Per Phase 4 §1 these are NOT tiered — register only at IRON.
    for (spec in Weapons.ALL) {
        if (!spec.isVanillaMapped && spec.ranged !in setOf(RangeKind.NONE, RangeKind.THROWN)) {
            register(spec, Tier.IRON)
        }
    }
    // 3. Tiered (melee + thrown) specs.
    for ((spec, tier) in entries) register(spec, tier)
}
```

**Imports to add:** `import com.dndweapons.catalog.RangeKind`

**Acceptance:**
- All 7 weapons (BLOWGUN, SLING, HAND_CROSSBOW, HEAVY_CROSSBOW, LONGBOW, MUSKET, PISTOL) appear in `BuiltInRegistries.ITEM` as `dndweapons:<id>` (registered at iron tier only).
- `AcquisitionCatalog.validate()` logs no "unknown weapon" errors for these 7 at startup.
- Existing 34-registered count assertion in `WeaponsTest.registeredCount` still passes (34 = 27 tiered base + 7 ranged registered iron-only — wait, total registered Items is now 27*3 + 7 ranged + 8 smithing = 96 Items. The `WeaponsTest.registeredCount` of 34 refers to `WeaponSpec` count, not Item count. No conflict.)

### P0-002 Generate 1.20.1 overlays for the 62 missing Phase 4 smithing recipes

**Root cause:** Only 34 base crafting recipes have 1.20.1 plural-folder overlays. The 54 smithing-transform + 4 component + 2 core + 2 template-assembly = 62 recipes are absent. The shared `recipe/` folder uses MC 1.21+ schema; 1.20.1 only reads `recipes/` plural.

**Files to create (62 total):**
```
versions/1.20.1/src/main/resources/data/dndweapons/recipes/
  <base>_diamond.json     (27 files: smithing_transform iron→diamond)
  <base>_netherite.json   (27 files: smithing_transform diamond→netherite)
  diamond_template_fragment.json     (component crafting)
  weapon_smithing_binding.json
  diamond_template_core.json
  netherite_template_fragment.json
  infernal_binding.json
  netherite_template_core.json
  diamond_weapon_upgrade_template_assemble.json   (smithing-table template assembly)
  netherite_weapon_upgrade_template_assemble.json
```

**Schema (1.20.1-format smithing_transform):**
```json
{
  "type": "minecraft:smithing_transform",
  "template_item": "dndweapons:diamond_weapon_upgrade_template",
  "base_item":     "dndweapons:longsword",
  "addition":      { "tag": "c:gems/diamond" },
  "result":        { "item": "dndweapons:longsword_diamond" }
}
```

Key differences from the shared 1.21+ format:
- `template_item` (not `template`)
- `base_item` (not `base`)
- `addition` as object `{ "tag": "..." }` not bare string `"#c:..."`
- `result.item` (not `result.id`)

**Implementation approach:** Add a `Phase4LegacyCodegen.kt` runnable main under `codegen/` that iterates `Weapons.ALL_TIERED` and emits 1.20.1-format JSONs. Pattern: mirror Phase4Codegen.kt but write to `versions/1.20.1/src/main/resources/data/dndweapons/recipes/` and use the 1.20.1 keys. Run once, commit output.

**Acceptance:**
- `find versions/1.20.1/src/main/resources/data/dndweapons/recipes/ -name '*.json' | wc -l` returns 96.
- A `:1.20.1:runGametest` smithing test passes: smithing `longsword + diamond + template` → `longsword_diamond`.

### P0-003 Convert shared smithing-transform recipes to object form (or add 1.21.4 + 1.21.11 overlays)

**Root cause:** Shared smithing-transform recipes use `"template": "<id>"` (bare string) and `"addition": "#c:..."` (hash-prefixed tag string). These are MC 1.21.5+ shorthand. 1.21.4 has no recipe overlay; if its recipe codec rejects shorthand, all 54 smithing upgrades fail to register on 1.21.4.

**Preferred fix (cleanest):** Convert the 54 shared smithing-transform recipes to universally-accepted object form:

```json
{
  "type": "minecraft:smithing_transform",
  "template": { "item": "dndweapons:diamond_weapon_upgrade_template" },
  "base":     { "item": "dndweapons:longsword" },
  "addition": { "tag": "c:gems/diamond" },
  "result":   { "id": "dndweapons:longsword_diamond" }
}
```

This format works on all 1.21.x (since 1.21.1 overlay already uses it). Drop the 1.21.1 overlay smithing recipes once shared format matches (per P2-001).

**Files to modify:** All 54 `src/main/resources/data/dndweapons/recipe/<weapon>_{diamond,netherite}.json` files. Optionally regenerate via Phase4Codegen.kt (modify its `emitSmithingTransformRecipes` to use object form).

**Acceptance:**
- `:1.21.4:runGametest` and `:1.21.11:runGametest` smithing tests pass.
- A grep for `"template": "dndweapons:` (bare string) in `src/main/resources/data/dndweapons/recipe/` returns zero matches.

### P1-006: Guard buildStructurePool against empty weapons list

```kotlin
private fun buildStructurePool(entry: StructureLoot): LootPool.Builder {
    val pool = LootPool.lootPool().setRolls(ConstantValue.exactly(1.0f))
    if (entry.weapons.isEmpty()) return pool   // NEW
    val items = entry.weapons.mapNotNull { WeaponLookup.byId(it, entry.tier) }
    if (items.isEmpty()) return pool           // NEW (combined with P2-010, P2-011 fix)
    val perWeaponWeight = maxOf(entry.chancePct / items.size, 1)
    for (item in items) { pool.add(LootItem.lootTableItem(item)...) }
    pool.add(EmptyLootItem.emptyItem().setWeight(maxOf(100 - entry.chancePct, 1)))
    return pool
}
```

Add validate() check: `if (loot.weapons.isEmpty()) errors += "structure $tableId: empty weapons list"`.
