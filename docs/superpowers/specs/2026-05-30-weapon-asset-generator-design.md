# Weapon Asset Generator — JSON-driven 16x16 pixel-art pipeline

**Date:** 2026-05-30
**Status:** Approved design, pending implementation plan
**Supersedes:** the external-LLM prompt workflow in the `weapon-asset-prompt` skill

## Problem

The current `weapon-asset-prompt` skill only emits a copy-paste text prompt for an
external image LLM (Gemini / GPT-image / Midjourney / SDXL). The human then has to
generate a 1024x1024 image elsewhere, nearest-neighbor downscale it to 16x16, and
chroma-key the magenta by hand. This is slow, non-deterministic, non-reproducible,
and depends on a third-party image model.

We want texture generation to be **deterministic and self-contained**: Claude
authors a JSON pixel-grid, and a converter that lives in this repo turns that JSON
into the final 16x16 RGBA PNG. Every weapon asset must be exactly 16x16.

## Goals

- Claude authors a JSON pixel-grid; a local Python program renders it to PNG. No
  external image LLM in the loop.
- Every produced texture is exactly 16x16 RGBA. Off-size or malformed input is
  rejected (hard-fail), never silently padded or cropped.
- Tier variants (iron / diamond / netherite) and the existing material overrides
  (leather / hardwood / lightwood / tan_leather / flintlock) are expressed as
  reusable color palettes, so one authored shape yields all of its tier PNGs.
- The existing domain knowledge — the 96-item Shape Lookup, the tier hex codes, and
  the per-weapon override mapping — is preserved, repurposed from "fill a prompt
  template" to "author a grid + choose palettes."

## Non-goals

- This effort does NOT hand-author all 96 shape grids. It ships the pipeline plus a
  single proven reference grid (`longsword`). The remaining 95 grids are authored
  on demand later by invoking the skill.
- No change to item model JSON, language files, loot, trades, or any non-texture
  asset.
- No reverse pipeline (PNG -> JSON). The 96 pre-existing PNGs are left as-is until a
  grid is authored to replace one.
- No image-LLM / MCP image-generation path. The pixel data is authored as JSON, not
  generated by a model-rendered image.

## Architecture

Three parts, one direction of data flow:

```
  Claude authors                  Python converter              Minecraft asset
  ------------------------        ------------------------      ----------------------
  art/shapes/<id>.json      +     scripts/render_textures.py -> textures/item/<id>.png
    (semantic char grid)          (resolve palette, validate,   (+ _diamond, _netherite
  art/palettes/<name>.json         render, write PNG)             when the weapon has tiers)
    (char -> hex)
```

Shape (the tier-independent char grid) is authored once per weapon. Tier palettes
(hex per semantic char) are small reusable files. The converter cross-products a
shape against the palettes named in that shape's `outputs` map.

## Source data layout

A new top-level `art/` directory holds the human/Claude-authored source. These are
build *inputs*, not shipped assets, so they live outside
`src/main/resources/`. The rendered PNGs continue to land in
`src/main/resources/assets/dndweapons/textures/item/`.

```
art/
  shapes/
    longsword.json          # one file per weapon (tier-independent grid)
  palettes/
    iron.json
    diamond.json
    netherite.json
    leather.json
    hardwood.json
    lightwood.json
    tan_leather.json
    flintlock.json
    # plus small per-tier override palettes that `extends` a base, e.g.
    leather_iron.json / leather_diamond.json / leather_netherite.json
    # (only created where an override body still varies a single accent char by tier)
```

### Shape file schema (`art/shapes/<id>.json`)

```json
{
  "id": "longsword",
  "size": 16,
  "outputs": {
    "longsword": "iron",
    "longsword_diamond": "diamond",
    "longsword_netherite": "netherite"
  },
  "rows": [
    "................",
    "..............B.",
    ".............Bb.",
    "............Bb..",
    "...........Bb...",
    "..........Bb....",
    ".........Bb.....",
    "........Bb......",
    ".......Bb.......",
    ".....ffBf.......",
    "....kkf.........",
    "...kk...........",
    "..kk............",
    ".kk.............",
    "kp..............",
    "p..............."
  ]
}
```

- `id` (string) — must match the filename stem.
- `size` (int) — must be `16`. Present for explicitness and future-proofing; the
  converter rejects any value other than 16.
- `outputs` (object) — maps **output PNG basename** (no extension) to a **palette
  name**. One entry per tier the weapon supports. A no-tier weapon (blowgun,
  templates) has a single entry, e.g. `{ "blowgun": "hardwood" }`.
- `rows` (array of 16 strings, each exactly 16 chars) — the pixel grid. `.` is
  transparent. Every other char is a **semantic** key resolved through the palette.

**Semantic char convention** (shape is tier-independent, so chars name a *role*, not
a color):

| char | role                         |
|------|------------------------------|
| `.`  | transparent                  |
| `B`  | primary highlight (blade/head/body highlight) |
| `b`  | primary midtone              |
| `s`  | primary shadow               |
| `w`  | wood midtone                 |
| `W`  | wood highlight               |
| `k`  | grip wrap / binding (dark)   |
| `f`  | metal fitting (crossguard/cap/pommel) |
| `p`  | pommel / butt accent         |
| `a`  | tier accent (ember speck, shard, inlay) |

The table is extensible; the authoritative list lives in the skill. The only hard
rule the converter enforces is: every non-`.` char used in `rows` MUST be defined in
the resolved palette.

### Palette file schema (`art/palettes/<name>.json`)

```json
{
  "extends": null,
  "colors": {
    "B": "#C8C8C8",
    "b": "#8B8B8B",
    "s": "#5A5A5A",
    "w": "#7C5A3A",
    "k": "#3E2A18",
    "f": "#6E6E6E",
    "p": "#6E6E6E",
    "a": "#6E6E6E"
  }
}
```

- `colors` (object) — semantic char -> `#RRGGBB` or `#RRGGBBAA` (or `null` to force
  transparent for that char).
- `extends` (string|null, optional) — name of a base palette to inherit from. The
  resolved palette is `base.colors` shallow-merged with this file's `colors` (this
  file wins). Inheritance chains are followed to a depth limit; cycles are rejected.

`extends` exists so override bodies that only change one accent cell per tier
(e.g. the whip's tip, the quarterstaff caps, the club shard inlay) don't duplicate a
full palette. Example: `leather_diamond.json` = `{ "extends": "leather", "colors":
{ "a": "#7DF2EE" } }`.

**All hex values are lifted verbatim from the current skill's tier blocks** — iron
`#C8C8C8 / #8B8B8B / #5A5A5A`, diamond `#7DF2EE / #4DD2D6 / #2A8FA6`, netherite
`#4A3A38 / #2E2422 / #1A1311`, leather/hardwood/lightwood/tan_leather/flintlock as
documented. No color rework.

## Converter — `scripts/render_textures.py`

Follows the conventions already established by `scripts/downscale_textures.py`:
`from __future__ import annotations`, module docstring, `argparse`, `REPO_ROOT`
derived from `__file__`, Pillow (`PIL.Image`), clear exit codes. Python 3.14 +
Pillow 12.1 are already present in the environment.

### Behavior

For each shape file (or just one with `--only`):
1. Load and validate the shape (see validation rules).
2. For each `(output_name, palette_name)` in `outputs`:
   a. Resolve the palette, following `extends`.
   b. Validate that every non-`.` char appearing in `rows` is defined in the
      resolved palette.
   c. Build a 16x16 RGBA image: map each char to its RGBA tuple (`.` and `null` ->
      `(0,0,0,0)`).
   d. Write `src/main/resources/assets/dndweapons/textures/item/<output_name>.png`
      via Pillow (`format="PNG"`, `optimize=True`).

### Validation (hard-fail, no silent fixes)

The converter exits non-zero with a precise message on any of:
- shape `size` != 16, or `rows` length != 16, or any row length != 16
- `id` does not match the filename stem
- a char in `rows` is not present in the resolved palette
- a palette hex value is not `#RRGGBB` / `#RRGGBBAA` (case-insensitive)
- `extends` names a missing palette, or forms a cycle, or exceeds the depth limit
- an `outputs` palette name has no corresponding palette file

Error format includes file, row, and column where applicable, e.g.:
`longsword.json row 4 col 9: char 'x' not defined in resolved palette 'iron'`.

### CLI

```
python scripts/render_textures.py            # render every shape in art/shapes/
python scripts/render_textures.py --only longsword
python scripts/render_textures.py --check     # validate all shapes+palettes, write nothing (CI-friendly)
```

`--check` performs full validation (size, palette resolution, char coverage, hex
format) without writing any PNG, and exits non-zero if anything is invalid. This
complements the existing `downscale_textures.py --check` (which asserts shipped PNGs
are 16x16).

## The skill (rewritten)

`weapon-asset-prompt` is renamed to **`weapon-asset-generator`**
(`.claude/skills/weapon-asset-generator/SKILL.md`). The directory is renamed and the
frontmatter `name`/`description` updated so triggers still fire on the same intents
("icon for the netherite maul", "generate the longsword texture", etc.).

### New responsibilities

Given a weapon id (+ optional tier filter), the skill:
1. Looks up the weapon's shape description in the preserved Shape Lookup table.
2. Authors a `art/shapes/<id>.json` grid using the semantic char convention, sizing
   the silhouette per the shape notes (diagonal bottom-left -> top-right, etc.).
3. Sets the `outputs` map to the correct palette per tier, using the override
   mapping (whip->leather, club->hardwood, musket/pistol->flintlock, ...).
4. Runs `python scripts/render_textures.py --only <id>`.
5. Reports the generated PNG path(s) and reminds the user to run
   `./gradlew chiseledBuild` to repackage the version jars.

### What is preserved from the old skill

- The full 96-item **Shape Lookup** table (now read as grid-authoring guidance).
- All **tier hex codes** (now the palette files' source of truth).
- The **per-weapon override mapping** (now drives each shape's `outputs` palettes).
- The orientation rules (sling vertical-V, blowgun/longbow exceptions, etc.).

### What is removed

- The 1024x1024 master prompt template and the GRID LOCK / NEGATIVE prose.
- The "paste into an image LLM" output format and worked prompt example.
- The manual downscale + chroma-key post-processing instructions.

## Deliverables for this effort

1. `art/palettes/` — all standard + override palette files, hex lifted from the
   current skill (including the small `extends` per-tier accent palettes needed by
   the override weapons).
2. `scripts/render_textures.py` — the converter with strict validation and
   `--only` / `--check` flags.
3. `art/shapes/longsword.json` — the single reference grid, rendering identical
   filenames to today's `longsword.png` / `longsword_diamond.png` /
   `longsword_netherite.png`.
4. `.claude/skills/weapon-asset-generator/SKILL.md` — the rewritten skill (renamed
   dir), with Shape Lookup, palette/override mapping, and the new author-grid-then-
   render workflow.

The remaining 95 shape grids are explicitly out of scope and authored later on
demand via the skill.

## Testing / verification

- `python scripts/render_textures.py --only longsword` produces three 16x16 RGBA
  PNGs; confirm with Pillow that each `.size == (16, 16)` and mode `RGBA`.
- `python scripts/render_textures.py --check` exits 0 on the committed `art/` set.
- Negative tests: a deliberately malformed shape (15 rows / a 17-char row / an
  undefined char / a bad hex / a missing palette / an `extends` cycle) each makes
  `--check` exit non-zero with the expected message.
- `python scripts/downscale_textures.py --check` still passes (all shipped textures
  remain 16x16).

## Open risks

- Authoring a recognizable weapon silhouette inside a 16x16 grid by hand is the hard
  part; the reference `longsword` grid sets the quality bar and the semantic-char
  vocabulary the other 95 will follow.
- Palette `extends` inheritance adds a little converter complexity; kept shallow
  (single-key overrides, depth-limited, cycle-checked) to stay simple.
